#1. Pendahuluan

  ##1.2 Latar Belakang

Aplikasi Manajemen Membership Gym dikembangkan untuk membantu proses pengelolaan data member, jadwal latihan, 
pembayaran membership, serta laporan administrasi secara digital dan terintegrasi.

Berdasarkan dokumen Software Requirement Specification (SRS) yang telah disusun, sistem ini memiliki dua aktor utama yaitu
admin dan member, dengan berbagai fitur seperti registrasi, login, pengelolaan jadwal, serta transaksi pembayaran.
         
  ##1.3 Ruang Lingkup

Ruang lingkup pengujian pada aplikasi ini mencakup seluruh fitur yang terdapat pada SRS, yaitu:

Pengujian pada sisi Member:
- Registrasi akun
- Login
- Melihat jadwal latihan
- Melihat status membership
- Melakukan pembayaran
- Perpanjangan membership

Pengujian pada sisi Admin:
- Login admin
- Mengelola data member (CRUD)
- Mengelola jadwal latihan
- Mengelola pembayaran
- Mencetak laporan
      
Selain itu, pengujian juga mencakup:

- Validasi input data
- Integrasi database (t_member, t_admin, t_jadwal, t_pembayaran)
- Pengujian fungsionalitas sistem secara keseluruhan

#2. Lingkungan Pengujian Perangkat Lunak

  ##2.1 Material Pengujian

Perangkat Keras (Hardware):
- Laptop/PC minimal RAM 8 GB
- Processor minimal Intel i5 atau setara
  
Perangkat Lunak (Software):
- Sistem Operasi: Windows 10/11
- Bahasa Pemrograman: PHP 
- Database: MySQL
- Web Browser: Google Chrome
- Tools Pengujian: Postman (untuk API testing)
  
Data Pengujian:
- Data member (nama, email, telepon, status)
- Data admin (username, password)
- Data jadwal (hari, jenis latihan, waktu)
- Data pembayaran (jumlah bayar, tanggal, keterangan)

  ##2.2 Teknik Pengujian

Teknik pengujian yang digunakan dalam sistem ini meliputi:

1). Black Box Testing
      
Digunakan untuk menguji fungsi sistem tanpa melihat struktur kode program. Fokus pada input dan output sesuai SRS.
      
2). Equivalence Partitioning (EP)
      
Digunakan untuk membagi data input ke dalam kelas valid dan tidak valid, seperti:
      
- Email valid vs tidak valid
- Password sesuai vs tidak sesuai
        
 3). Boundary Value Analysis (BVA)
      
Digunakan untuk menguji batas nilai input, seperti:
          
- Panjang karakter nama
- Panjang password
  
4). Validation Testing
      
Digunakan untuk memastikan setiap field input memenuhi aturan seperti required, format, dan tipe data.
      
5). Functional Testing
      
Dilakukan untuk memastikan seluruh fitur berjalan sesuai kebutuhan sistem yang telah didefinisikan dalam SRS.

#4. Hasil Pengujian 

Berdasarkan hasil pengujian yang telah dilakukan, diperoleh beberapa kesimpulan sebagai berikut:

Hasil Umum:
- Seluruh fitur utama sesuai dengan SRS telah berjalan dengan baik
- Sistem mampu menangani input valid dan tidak valid dengan benar
- Proses CRUD pada data member, jadwal, dan pembayaran berjalan sesuai harapan
- Sistem berhasil menampilkan laporan sesuai data yang tersimpan
  
Hasil Validasi:
- Validasi input seperti email, password, dan field wajib telah berfungsi dengan baik
- Error handling sudah mampu memberikan feedback kepada pengguna
  
Temuan (Bug):
- Ditemukan bug pada proses login ketika field password kosong (sesuai SRS)
- Bug telah diperbaiki (status: fixed)
  
 Evaluasi Sistem:

Secara keseluruhan, aplikasi telah memenuhi kebutuhan fungsional yang telah didefinisikan dalam dokumen SRS. Sistem dinilai stabil dan siap digunakan dengan beberapa perbaikan minor jika diperlukan.
